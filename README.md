## 📁 Структура репозитория

В репозитории представлены два основных проекта:

### 1. `docker-example/` (Контейнеризация)
Пример грамотной упаковки Python-приложения в Docker-образ и локального запуска вместе с базой данных (Redis) через `docker-compose`.

**Ключевые особенности сборки:**
* **Multi-stage build:** Сборка зависимостей вынесена в отдельный этап (`builder`), что позволяет значительно уменьшить итоговый размер образа и не тянуть лишние сборочные утилиты в продакшн.
* **Security Best Practices:** Приложение запускается от имени непривилегированного пользователя (`appuser`), а не от `root`, что минимизирует риски безопасности.
* **Изоляция и кеширование:** Использование флагов `--no-cache-dir` при установке зависимостей.
* **Docker Compose:** Настроен `docker-compose.yml` для автоматического поднятия веб-приложения и Redis в единой изолированной сети (`app-network`).

### 2. `helm-example/` (Оркестрация в Kubernetes)
Базовый Helm-чарт для деплоя stateless веб-приложения в кластер Kubernetes.

**Что реализовано в чарте:**
* **Гибкая конфигурация:** Все ключевые параметры (теги образов, количество реплик, порты) вынесены в `values.yaml` для удобного переиспользования чарта в разных окружениях (dev/prod).
* **Управление ресурсами:** В шаблоне деплоя (`deployment.yaml`) прописаны `resources` (limits и requests) для CPU и RAM, что предотвращает "поедание" ресурсов кластера одним подом.
* **Сетевая доступность:** Шаблон `service.yaml` для маршрутизации внутреннего трафика к подам.

## 🚀 Как запустить примеры локально

### Запуск Docker Compose:
```bash
cd docker-example
docker-compose up -d --build
```

---

После запуска приложение будет доступно на порту 5000, а Redis — на стандартном 6379.
Проверка Helm-чарта (Рендеринг манифестов):

Чтобы посмотреть, какие манифесты сгенерирует Helm перед деплоем в кластер, используйте команду:

```bash
cd helm-example
helm template my-release ./my-app
```
